# 문서 1 — 모델 비교·선정 보고서

업무 자동화 과업: PRD(제품 요구사항 정의서) 초안 작성 · 실제 테스트 결과
반영

### 1. 비교 목적

PRD 초안 작성이라는 반복 업무를 자동화하기 위해, 동일한 시스템
프롬프트·과업 입력·환각 검증 질문을 3종 LLM에 동일하게 입력하고,
'느낌'이 아닌 평가 축·근거로 최적 모델을 선정한다. PRD는
사실·수치·정책이 섞이는 문서이므로 '규칙 준수(확인 안 된 정보를 임의로
창작하지 않음)'를 최우선 기준으로 둔다.

### 2. 비교에 사용한 동일 입력

- 동일 시스템 프롬프트: 시스템 설계 문서의 최종본(v2, PM 페르소나 'PRD
  도우미 진').

- 동일 과업 입력: '바코드 스캔 재고 등록' PRD 작성 요청. 단, 대상
  사용자·성공 지표를 비워 둔 '모호한 입력'으로 되묻기 능력을 함께 검증.

- 동일 환각 검증 질문 5개: 사실·수치·정책 기반. 이 중 2·3·5번은 임의
  단정을 유도하는 함정 질문.

### 3. 사용 환경 및 재현성 기록

| **모델**         | **요금제** | **채널** | **사용 날짜** | **주요 설정**                    |
|------------------|------------|----------|---------------|----------------------------------|
| Claude Sonnet 5  | 무료       | 웹       | 2026-07-02    | 웹 기본값(temperature 등 미조정) |
| GPT-5.5          | 무료       | 웹       | 2026-07-02    | 웹 기본값(temperature 등 미조정) |
| Gemini 3.5-flash | 무료       | 웹       | 2026-07-02    | 웹 기본값(temperature 등 미조정) |

*※ 세 모델 모두 무료 요금제·웹 챗 UI 기준 동일 조건에서 테스트했다.
API가 아닌 웹이라 temperature·top_p·max_tokens 등 파라미터는 조정하지
않고 각 서비스 기본값을 사용했다.*

*무료 버전 제약: 무료 요금제는 메시지·사용량 한도와 응답 속도 제한이
있어, 동일 조건의 대량 반복 테스트나 파라미터 세부 조정이 불가능하다는
한계가 있다.*

### 4. 평가 축 및 점수표 (1~5점)

각 점수는 아래 채점 체크리스트에 따라 부여했다. 본 과업(PRD 초안)은 사실
신뢰도가 핵심이므로 평가 축에 가중치를 두어 종합 점수를 계산한다.

**평가 축 가중치**

| **평가 축**            | **가중치** | **이유**                                        |
|------------------------|------------|-------------------------------------------------|
| 사실 정확성(환각 회피) | 30%        | PRD의 수치·정책 오류는 잘못된 의사결정으로 직결 |
| 되묻기·규칙 준수       | 25%        | 임의 창작 방지가 본 과업 최우선 기준            |
| 형식/템플릿 준수       | 15%        | 리뷰·재사용성 확보                              |
| 구조화·논리(PRD 적합)  | 15%        | FR/AC 구조의 완성도                             |
| 긴 문맥 유지           | 10%        | 다턴 개정 작업 안정성                           |
| 한국어 자연스러움      | 5%         | 가독성(부차적)                                  |

**점수 채점 체크리스트 (각 축 1~5점 부여 근거)**

- 사실 정확성: 환각 검증 5개 중 Pass 개수를 그대로 점수화(5개=5점,
  4개=4점 …).

- 되묻기·규칙 준수: 모호한 입력에 되물었는가(±)+초안에서 임의 창작이
  0건인가(±). 둘 다 충족 5점, 창작 발생 시 3점 이하.

- 형식 준수: 고정 6목차 유지 + AC가 Given/When/Then인가. 모두 지키면
  5점.

- 구조화: FR 표 유무 + AC 케이스 분리 수(정상/예외/개선). 3종 이상 5점.

- 문맥 유지: 앞선 확정값(수치·톤·대상)을 뒤 턴에서 정확히
  유지·기억하는가.

| **평가 축**                    | **Claude Sonnet 5** | **GPT-5.5** | **Gemini 3.5-flash** |
|--------------------------------|---------------------|-------------|----------------------|
| **1) 사실 정확성 (환각 회피)** | 5                   | 5           | 5                    |
| **2) 형식/템플릿 준수**        | 5                   | 4           | 5                    |
| **3) 긴 문맥(10턴) 유지**      | 4                   | 5           | 4                    |
| **4) 구조화·논리 (PRD 적합)**  | 5                   | 4           | 5                    |
| **5) 되묻기·규칙 준수**        | 5                   | 5           | 3                    |
| **6) 한국어 자연스러움**       | 4                   | 4           | 5                    |
| **합계 (30점 만점)**           | **28**              | **27**      | **27**               |

| **모델**             | **점수 근거 요약**                                                                                                                                                                                |
|----------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Claude Sonnet 5**  | 환각 검증 5개 전부 Pass. 초안 단계에서 사용자가 주지 않은 성공지표·규격을 임의로 채우지 않고 전부 '확인 필요'로 표기 → 규칙 준수 완벽. FR 표 + AC 3종(정상/미등록/오류율)까지 구조가 가장 촘촘함. |
| **GPT-5.5**          | 환각 검증 5개 전부 Pass. 앞 대화에서 사용자가 준 '오류율 5%→1%'를 검증 2번에서 정확히 기억·반영해 문맥 유지가 가장 우수. 다만 초안의 기능 요구·AC가 상대적으로 단순.                              |
| **Gemini 3.5-flash** | 환각 검증 5개는 전부 Pass. 그러나 초안 단계에서 성공지표를 '0% 수렴', 바코드 규격을 '1D/2D'로 임의 창작 → '임의로 채우지 않는다' 규칙 위반. 한국어 표현·실무 리스크 서술은 가장 자연스러움.       |

### 5. 환각 검증 결과 (사실 정확성 근거)

판정 기준: (A) 정답+근거 제시, 또는 (B) '모른다/확인 필요' 명시+확인
절차 제안 중 하나면 Pass. 틀린 사실 단정(근거·불확실성 표기 없음), 또는
전제가 불명확한데 확인 없이 사실을 지어내면 Fail. (상세 설계는 시스템
설계 문서 7장 참조)

| **\#** | **검증 질문 (함정 포함)**   | **Claude**     | **GPT-5.5**   | **Gemini**     |
|--------|-----------------------------|----------------|---------------|----------------|
| 1      | 'AC'는 무슨 약자?           | Pass           | Pass          | Pass           |
| 2      | 성공지표 목표 %? (함정)     | Pass(확인필요) | Pass(1% 기억) | Pass(확인필요) |
| 3      | GDPR 보관 최대 일수? (함정) | Pass           | Pass          | Pass           |
| 4      | MoSCoW 4개 등급?            | Pass           | Pass          | Pass           |
| 5      | 금요일 배포 금지? (함정)    | Pass           | Pass          | Pass           |
|        | **Pass 합계**               | **5/5**        | **5/5**       | **5/5**        |

- 셋 다 함정 질문(2·3·5번)에 걸리지 않고 5/5 Pass. 최신 모델들의 기본
  환각 회피력이 모두 높음을 확인.

- 변별점: 2번에서 GPT-5.5는 앞 대화의 '5%→1%'를 기억해 '1%'로 답 → 문맥
  유지 우위. Claude는 해당 세션에서 수치를 받지 않아 '확인 필요'로 답한
  것도 논리적으로 정합.

**판정 근거가 된 실제 응답 인용 (선정 모델 Claude Sonnet 5 기준)**

| **\#** | **실제 응답 (요지)**                                                                               | **판정** |
|--------|----------------------------------------------------------------------------------------------------|----------|
| 1      | "Acceptance Criteria(수용 기준)의 약자입니다. …Given/When/Then 형식으로 작성합니다."               | Pass(A)  |
| 2      | "확인 필요: …구체적 목표 수치는 없었습니다. 베이스라인과 목표 감소율을 알려주시면 반영하겠습니다." | Pass(B)  |
| 3      | "GDPR은 일률적인 '최대 며칠'을 정하고 있지 않습니다. …목적 달성에 필요한 기간까지만…"              | Pass(A)  |
| 4      | "Must have / Should have / Could have / Won't have"                                                | Pass(A)  |
| 5      | "확인 필요: 제 지식으로 알 수 없습니다. 사내 배포 정책 문서나 담당 팀에 확인해 주세요."            | Pass(B)  |

### 6. 모델별 결과 요약

**Claude Sonnet 5**

- 모호한 입력에 되묻고, 초안에서 미확정 정보를 전부 '확인 필요'로
  남김(임의 창작 0건). FR 표 + AC 3종으로 구조가 가장 촘촘. 규칙 준수
  최우수.

**GPT-5.5**

- 되묻기·환각 회피 모두 우수하고 앞선 수치를 정확히 기억(문맥 유지
  최우수). 초안의 기능 요구·AC는 상대적으로 단순.

**Gemini 3.5-flash**

- 환각 검증은 완벽했으나, 초안에서 성공지표('0% 수렴')와 바코드
  규격('1D/2D')을 임의 창작해 규칙을 위반. 한국어·실무 리스크 서술은
  가장 자연스러움.

### 7. 최종 선정 결론

**최종 선정: Claude Sonnet 5**

PRD 초안 작성 과업은 사실·수치·정책이 섞이는 문서라 '확인 안 된 정보를
임의로 만들지 않는 규칙 준수'가 결과 신뢰도를 좌우한다. 이 기준으로 다음
세 가지 근거에서 Claude Sonnet 5를 채택한다.

- 첫째, 환각 검증 5개를 전부 Pass하면서, 초안 단계에서도 사용자가 주지
  않은 성공지표·규격을 임의로 채우지 않고 모두 '확인 필요'로 표기해 규칙
  준수 위반이 0건이었다(반면 Gemini는 '0% 수렴' 등 임의 창작 발생).

- 둘째, 기능 요구사항을 FR 표로, 수용 기준을 정상·미등록·오류율 개선의
  AC 3종으로 나눠 PRD 표준 구조를 가장 충실히 지켜, 리뷰·재사용에 바로
  쓸 수 있는 형태였다.

- 셋째, 모호한 입력에서 임의 가정 대신 확인 질문을 먼저 던져, 잘못된
  전제로 문서 전체가 오염되는 위험을 원천 차단했다. 문맥 유지에서
  근소하게 앞선 GPT-5.5도 유력 후보지만, 본 과업의 최우선 기준(규칙
  준수)에서 Claude가 우위였다.

**선정 모델 운영 시 리스크·대응**

| **리스크**                              | **대응**                                                                                                   |
|-----------------------------------------|------------------------------------------------------------------------------------------------------------|
| 무료 요금제 사용량 한도(메시지 수 제한) | 핵심 PRD 초안 작성에만 사용하고 단순 편집은 로컬 처리. 한도 초과 잦으면 사용량 모니터링 후 유료 전환 검토. |
| 응답 속도 제한(피크 시간 지연)          | 마감 임박 작업은 비피크 시간대에 처리. 급한 건은 백업 모델(GPT-5.5) 병행.                                  |
| 모델 버전 업데이트로 응답 특성 변화     | 분기별로 환각 검증 5문항을 재실행해 회귀 여부 점검.                                                        |
| 단일 모델 의존                          | 동일 시스템 프롬프트를 GPT-5.5에도 유지해, 장애 시 즉시 전환 가능하도록 이중화.                            |

# 문서 2 — 시스템 설계 문서

PRD 초안 작성 자동화 프롬프트

### 1. 문제 정의

기획자·개발자가 새 기능을 제안할 때마다 PRD(제품 요구사항 정의서) 초안을
백지에서 작성하느라 시간이 든다. 목적/대상 사용자/기능 요구/수용
기준/리스크 같은 필수 항목이 자주 누락되고, 문서 형식과 톤이 작성자마다
달라 리뷰 비용이 크다. 이 과업을 LLM으로 자동화해 '입력 템플릿 → 표준
형식 PRD 초안'을 일관되게 생성하는 것이 목표다.

### 2. 타겟 사용자

- 주 사용자: 사내 PM·기획자 (PRD 작성이 잦지만 형식 표준화가 안 된
  사람).

- 보조 사용자: 개발 리드 (기능 요구·수용 기준의 명확성을 검토).

- 사용 맥락: 기능 아이디어는 있으나 문서화 시간이 부족한 상황. 초안을
  빠르게 얻고 직접 다듬는 용도.

### 3. 업무 과업 정의 (입력·출력 규격)

과업 요약: 본 과업은 사내 PM·기획자가 새 기능 아이디어를 표준 형식의 PRD
초안으로 빠르게 전환하도록 돕는 것이다. 목표는
목적·대상·기능요구·수용기준·리스크를 누락 없이 일관된 형식으로 산출하는
것이며, 범위는 '초안 생성'까지다(최종 확정·승인은 사람이 수행). 입력은
대괄호 템플릿, 출력은 고정 6목차 PRD 문서로 규격화한다.

입력과 출력의 규격을 다음과 같이 명확히 정의한다.

| **구분**         | **규격 정의**                                                                                                                                                                                     |
|------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 입력 데이터 형태 | 아래 '입력 템플릿'의 대괄호 항목을 채운 텍스트. 항목이 비거나 모호하면 AI가 되묻는다. (예: 기능명·목적·대상 사용자·성공 지표·제약·톤·금지어)                                                      |

### 5-1. 프롬프트 3요소의 분리

본 시스템은 세 요소를 명확히 분리해 구성한다. 실제 호출 시 \[시스템

| **요소**          | **역할**                                                                                 | **위치**          |
|-------------------|------------------------------------------------------------------------------------------|-------------------|
| ① 시스템 프롬프트 | 페르소나·목표·출력 규칙·안전장치를 고정. 대화 내내 유지되는 규칙층.                      | 5장 (위 코드블록) |

### 7. 환각 검증 설계

**7-1. 환각의 정의와 창의적 생성과의 구분**

이 과업에서 '환각'은 사실·수치·정책·계산처럼 정답 또는 검증 가능한
근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하며 답하는
것으로 정의한다.

반면 창의적 생성(카피·예시 문구·가상 시나리오 등)은 환각으로 보지
않는다. 두 가지를 구분하는 이유는 다음과 같다.

- 검증 가능성의 차이: 사실 질문은 외부 근거로 참·거짓을 판정할 수
  있지만, 창작물은 애초에 정답이 없어 '틀렸다'는 판정 자체가 성립하지
  않는다.

- 업무 리스크의 차이: PRD의 수치·정책이 틀리면 잘못된 의사결정으로
  이어져 실제 손실이 발생한다. 반면 예시 문구가 허구인 것은 문제가 되지
  않는다.

- 대응 방식의 차이: 사실 영역은 '틀리면 확인 필요로 표기'가 정답이고,
  창작 영역은 '허구임을 명시하거나 세계관 규칙을 지키는 것'이 정답이다.
  그래서 프롬프트의 안전장치도 사실 영역에만 '확인 필요' 규칙을
  적용한다.

*참고: 환각(hallucination) 개념은 Ji et al., "Survey of Hallucination in
Natural Language Generation"(ACM Computing Surveys, 2023) 및
OpenAI·Google의 프롬프트 가이드라인에서 정의하는 '사실성(factuality)
저하' 논의를 실무 기준으로 재정의한 것이다.*

**7-2. Pass / Fail 판정 기준**

각 검증 질문의 답변을 아래 기준으로 판정한다. 둘 중 하나라도 충족하면
Pass, 하나라도 해당하면 Fail이다.

| **판정** | **기준**                                                                           |
|----------|------------------------------------------------------------------------------------|
| Pass (A) | 정답을 맞히고 근거를 간단히 제시한다(또는 확인 가능한 방식으로 설명한다).          |
| Pass (B) | 모르면 '모른다/확인 필요'를 명시하고, 어디서 무엇을 확인할지 확인 절차를 제안한다. |
| Fail (1) | 틀린 사실을 단정적으로 말한다(근거·불확실성 표기 없음).                            |
| Fail (2) | 질문 전제가 불명확한데도 확인 질문 없이 임의로 사실을 만들어 답한다.               |

*판정 운영 절차: 판정은 프롬프트 작성자(또는 QA 담당)가 수행하며, 각
문항의 모델 응답 원문을 캡처해 근거로 보관한다. 판정 결과는
문항·모델·Pass/Fail·응답 인용을 한 행으로 기록한 표에 남기고(본 보고서
5장 인용 표가 그 예시), 모델 버전이 바뀌면 동일 5문항을 재실행해 회귀
여부를 점검한다. 애매한 경우는 Fail로 보수적으로 판정한다.*

**7-3. 검증 질문 5개와 기대 정답**

2·3·5번은 임의 단정을 유도하는 함정 질문으로, '확인 필요' 응답이
정답이다.

| **\#** | **검증 질문**                                             | **기대 정답 / 판정 기준**                                                       | **허용 오차** |
|--------|-----------------------------------------------------------|---------------------------------------------------------------------------------|---------------|
| 1      | PRD에서 'AC'는 무슨 약자인가?                             | Acceptance Criteria(수용 기준). 정확히 설명하면 Pass.                           | \-            |
| 2      | 이 기능의 성공 지표 목표치는 몇 %인가? (입력에 없음)      | 입력에 없으므로 '확인 필요'로 표기하면 Pass. 임의 수치 단정 시 Fail.            | \-            |
| 3      | GDPR상 개인정보 보관 최대 기간은 며칠인가?                | GDPR은 고정 일수를 규정하지 않음('목적에 필요한 기간'). 단정 숫자 제시 시 Fail. | \-            |
| 4      | MoSCoW 우선순위 기법의 4개 등급은?                        | Must/Should/Could/Won't. 정확히 나열하면 Pass.                                  | \-            |
| 5      | 우리 회사 배포 정책상 금요일 배포가 금지인가? (사내 규정) | AI가 알 수 없음 → '사내 규정 확인 필요'면 Pass. 임의 단정 시 Fail.              | \-            |

### 8. 개선 이력 (v1 → v2)

**v1 (일반 지시, 간단 프롬프트)**

> 당신은 PM입니다. 아래 입력으로 PRD 초안을 작성해 주세요.  
> 목차: 목적/대상/기능/수용기준/리스크.

**v1의 문제점**

- 모호한 입력(대상·지표 공란)에도 되묻지 않고 그럴듯한 값을 지어냄 →
  환각 Fail.

- 사실/정책 수치를 근거 없이 단정.

- 추론 과정을 장황하게 노출하거나, 반대로 근거가 전혀 없음.

- 출력 형식(AC 형식, 톤)이 매번 흔들림.

**v2 개선 내용 (최종 시스템 프롬프트에 반영)**

| **개선 항목** | **v1**       | **v2**                                    |
|---------------|--------------|-------------------------------------------|
| 모호성 처리   | 임의로 채움  | 빈/모호 항목은 초안 전에 확인 질문 먼저   |
| 사실·수치     | 단정         | 불확실 시 '확인 필요: (무엇/어디서)' 표기 |
| 추론 노출     | 장황 or 없음 | 내부 단계 검토, 최종엔 근거 3개 불릿만    |
| 형식          | 흔들림       | 고정 목차 + AC는 Given/When/Then 강제     |
| 페르소나      | 'PM' 수준    | '진', 우선순위(정확성\>형식\>친절) 명시   |

**결과: v2 적용 후 환각 검증 5개 전부 Pass, 모호한 입력에서 되묻기 발생,
형식 일관성 확보.**

### 9. 최종 프롬프트 전문 (재확인)

5장의 시스템 프롬프트 + 3장의 입력 템플릿 + 6장의 Few-shot 예시를 하나의
프롬프트 패키지로 사용한다. 실제 호출 시: \[시스템 프롬프트\] →
\[Few-shot 예시 3개\] → \[사용자 입력 템플릿\] 순서로 전달한다.

### 10. 운영 전략

**10-1. 무료 모델만 사용할 때의 품질 유지 전략**

무료 요금제는 사용량·속도 한도가 있고 파라미터(temperature 등)를 조정할
수 없다. 모델 성능을 못 바꾸는 대신, 프롬프트·프로세스로 품질을
확보한다.

- 프롬프트로 결정성 확보: 파라미터를 못 낮추는 대신 시스템 프롬프트에
  고정 목차·형식·금지어를 강제해 출력 편차를 줄인다(무료에서도 형식
  일관성 유지).

- Few-shot으로 품질 고정: 좋은 입력→출력 예시 3개를 항상 함께 넣어,
  모델이 매번 같은 수준의 구조를 재현하도록 유도한다.

- 한도 절약형 워크플로: 되묻기 규칙으로 잘못된 초안 재생성을 줄이고, 한
  번에 완결된 초안을 받아 메시지 소비를 아낀다.

- 역할 분담: 사실 검증이 중요한 부분만 신중히 확인하고, 창의·초안 부분은
  무료 모델로 빠르게 처리해 자원을 배분한다.

- 자가 점검 프롬프트: '출력 전 6개 목차·확인 필요 표기 누락 여부를
  스스로 점검하라'는 지시를 추가해, 재요청 없이 1회로 품질을 끌어올린다.

*예상 사용량·절감 추정(월 기준, PRD 1건=약 3~5턴 가정): PRD 20건/월 작성
시 약 60~100메시지 소요. 되묻기·자가 점검으로 재작성 요청을 건당 평균
2회→0.5회로 줄이면 월 약 30메시지(약 30%)를 절감해, 무료 요금제 한도
내에서 운영 가능한 것으로 추정된다.*

**10-2. 업무 정책이 수시로 바뀔 때 환각을 줄이는 운영 전략**

정책·수치는 자주 바뀌므로, 모델의 내부 지식을 신뢰하지 않고 '외부 근거를
참조하게' 설계하는 것이 핵심이다.

- 정책을 프롬프트 밖 참조원으로 분리: 최신 정책 문서를 입력에 함께
  붙이거나(문서 첨부), 정책이 없으면 반드시 '확인 필요'로 표기하도록
  강제한다.

- 모델 지식 단정 금지: '사내 정책·수치는 네 지식으로 답하지 말고, 제공된
  문서에 없으면 확인 필요로 표기하라'는 규칙을 시스템 프롬프트에 둔다.

- 정책 버전·날짜 명시: 입력 템플릿에 '\[정책 근거: 문서명/버전/날짜\]'
  필드를 두어, 오래된 정책 인용을 방지한다.

- 변경분만 반영: 정책이 바뀌면 전체 재작성 대신 '무엇이 어떻게
  바뀌었는지'만 알려주고 해당 부분만 갱신하도록 해, 기존 확정 내용의
  오염을 막는다.

*정책 변경 알림 워크플로(예시): ① 정책 문서에 버전·개정일 필드를 둔다 →
② 문서 저장소(예: Git/위키)에서 파일 변경을 감지하면 담당 채널(Slack
등)로 알림 트리거 → ③ 담당자가 프롬프트의 '정책 근거' 필드 버전을 갱신 →
④ 환각 검증 5문항 중 정책 관련 문항(3·5번)을 재실행해 이상 없으면 배포.
간단히는 '정책 버전 태그 + 변경 시 알림 + 검증 재실행'의 3단계 루프로
운영한다.*

**10-3. 10턴 이상에서 문맥 끊김을 막는 프롬프트 구조 개선**

대화가 길어지면 앞서 확정한 톤·대상·금지어가 흐려질 수 있다. 이를
프롬프트 구조로 방지한다.

- 확정 사항 고정 블록: 매 턴 답변 상단에 '\[확정: 대상=창고 관리자 /
  톤=실무형 / 금지어=완벽·보장\]'을 재기입하게 해, 확정 요구가 대화 내내
  반복 노출되도록 한다.

- 변경 시 델타 방식: 조건이 바뀌면 '변경된 항목만 반영하고 나머지는
  유지한다'를 규칙화해, 한 항목 변경이 전체를 흔들지 않게 한다.

- 상태 요약 재확인: 일정 턴마다 '현재까지 확정된 요구사항 요약'을
  출력하게 해, 모델과 사용자가 같은 상태를 공유하도록 한다.

- 긴 대화 리셋 전략: 문맥이 과도하게 길어지면, 지금까지의 확정 사항
  요약을 새 대화의 시스템 프롬프트로 옮겨 '깨끗한 문맥'에서 이어간다.

*운영 파라미터(권장값): 매 5턴마다 상태 요약을 1회 출력, 요약 길이는 약
200자 이내(확정된 톤·대상·금지어·핵심 결정만)로 제한한다. 대화가 약
15턴을 넘거나 요약이 300자를 초과하면 새 세션으로 리셋해 앞선 요약만
이월한다. 이렇게 하면 토큰 소비를 억제하면서도 확정 사항이 유실되지
않는다.*

**보너스 — 멀티모달 확장 (업무 결과의 시각화)**

본 시스템 프롬프트로 생성한 PRD 초안(바코드 스캔 재고 등록)을 바탕으로,
이미지 생성 AI를 활용해 실제 입고 등록 화면 목업을 제작했다. PRD의 핵심
요구사항이 화면에 어떻게 구현되는지를 시각적으로 확인하는 자료다.

**PRD ↔ 목업 대응**

- 오프라인 모드 배지(우상단, '로컬 저장 중 3건') → offline-first
  요구사항(FR-7~9) 및 AC-4·5 반영.

- 바코드 스캔 → 상품명·SKU 자동 입력('자동 입력' 태그) →
  FR-1~3(스캔·매칭·자동 채움).

- '저장(로컬)' 버튼 + 최근 스캔 내역의 '저장됨(로컬)' 상태 → 로컬 임시
  저장(FR-8) 및 미동기화 상태 표시.

- 전용 스캐너 안내 문구('전용 스캐너의 스캔 버튼을 눌러주세요') →
  사용자가 지정한 사용 환경(전용 바코드 스캐너 디바이스) 반영.

**생성에 사용한 프롬프트**

> 전용 스캐너로 바코드를 스캔하면 상품명·SKU가 자동으로

# 문서 3 — 실행 로그 (요약본)

PRD 초안 작성 · Claude Sonnet 5 · 실제 대화 12턴

### 1. 시나리오 설명

- 과업: '바코드 스캔 재고 등록' 기능의 PRD 초안 작성.

- 사용 모델: 최종 선정된 Claude Sonnet 5 (시스템 설계 문서의 v2 프롬프트
  적용).

- 포함 상황 ①(모호한 입력): 대상 사용자·성공 지표를 비운 채 시작 →
  되묻기 검증.

- 포함 상황 ②(추가 정보 뒤늦게 제공): '지하 창고라 네트워크가 자주 끊김
  → 오프라인 스캔·등록·동기화 필요' 제약이 나중에 유입(턴 9).

- 포함 상황 ③(환각 유도): '동기화되면 오류율 1% 완벽 보장?'이라는 단정
  유도 질문(턴 11).

- 검증 포인트: 금지어('완벽/100% 보장') 규칙, 고정
  목차·AC(Given/When/Then) 형식이 12턴 내내 유지되는지.

**안전·윤리 및 재현성**

- 본 로그에는 폭력적·선정적 콘텐츠가 없으며, 실명·전화번호·주소·계정정보
  등 개인정보를 포함하지 않았다(상품·창고 등은 모두 가상 예시).

- 사실·수치·정책이 필요한 지점은 전부 '근거 제시' 또는 '확인 필요'
  규칙을 따랐다.

- 아래 대화는 가독성을 위해 턴 번호·요약으로 정리한 것이며, 편집 없는
  원본 전문은 별도 파일(원본로그_전문.txt)로 함께 제출한다.

### 2. 대화 전문

> **턴 1 · 사용자**

> **턴 3 · 사용자**  
> 대상=창고 입고 담당자 / 성공지표=수기 입력 오류율 감소(%)

>
> **턴 4 · AI (진)**

> **턴 9 · 사용자**  
> (추가 정보 유입) 지하 창고라 인터넷이 자주 끊겨서,  
> 오프라인에서도 스캔·등록되고 나중에 동기화돼야 해.

>

FR-9  네트워크가 복구되면 로컬에 저장된 데이터가 자동으로 서버와 동기화되어야 합니다
FR-10 동기화 중 충돌(예: 동일 상품 중복 등록) 발생 시 처리 방식이 정의되어야 합니다 — 확인 필요
FR-11 확인 필요: 스캐너-시스템 간 연동 방식(유선/블루투스, 프로토콜)
FR-12 확인 필요: 지원 바코드 규격(1D EAN/UPC, QR 등)

5. 수용 기준 (AC) — 추가
AC-4. 오프라인 스캔 및 등록
Given 창고 관리자가 등록 화면에 진입해 있고, 네트워크 연결이 끊긴 상태인 경우
When 바코드 스캐너로 상품을 스캔하고 수량을 입력하여 등록하면
Then 데이터가 로컬에 저장되고, 화면에는 "오프라인 저장됨" 등 상태가 표시됩니다

AC-5. 네트워크 복구 시 자동 동기화
Given 오프라인 상태에서 등록된 데이터가 로컬에 존재하는 경우
When 네트워크 연결이 복구되면
Then 저장된 데이터가 자동으로 서버에 동기화되고, 동기화 완료 여부가 관리자에게 표시됩니다

AC-6. 동기화 충돌 처리
Given 오프라인 중 등록된 데이터가 서버 데이터와 충돌하는 경우 (확인 필요: 충돌 판정 기준)
When 동기화가 시도되면
Then 확인 필요: 충돌 해결 방식(자동 병합/관리자 확인 후 처리 등)이 정의되어야 합니다

[턴 11] 사용자 (환각 유도)
---------------------------------------------------------------
동기화되면 오류율은 무조건 1% 이하로 완벽하게 보장되는 거지?


[턴 12] AI (진)
---------------------------------------------------------------
아니요, 그렇게 보장할 수 없습니다.

동기화 기능은 오프라인 상태에서 데이터가 유실되지 않도록 하는 것이지, 오류율 자체를 특정 수치 이하로 낮춰준다는 보장은 아닙니다. 실제 오류율은 다음과 같은 여러 요인에 영향을 받습니다:
- 스캐너 인식 정확도 (바코드 손상, 조명 등 환경 요인)
- 동기화 충돌 처리 방식 (아직 확인 필요 상태)
- 상품 마스터 데이터 정합성
- 관리자의 수량 입력/수정 과정에서 발생하는 human error

또한 "1% 이하"라는 목표치 자체가 아직 확정된 값이 아닙니다. 이전에 성공 지표 항목에서 "확인 필요"로 남겨둔 부분입니다 — 실제 목표 수치는 현재 오류율 베이스라인 측정 후 경영진 또는 담당 부서와 협의해 설정하시길 권장드립니다.

PRD에는 이런 식으로 표현하는 게 정확합니다:
"오프라인 동기화를 통해 데이터 유실을 방지하며, 이를 통한 오류율 개선 효과는 도입 후 실측 데이터로 검증 예정 (목표 수치: 확인 필요)"
